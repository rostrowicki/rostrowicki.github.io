Entity Framework

General Concepts

Entity Framework is an object-relational mapper (ORM) that enables .NET developers to work with a database using domain-specific objects. It eliminates the need for most of the data-access code that developers usually need to write.

Here’s a high-level overview of how Entity Framework works:

You design your domain model classes in your .NET application, with each class representing a table in your database.

You use Entity Framework to create a mapping between your domain model classes and the corresponding tables in the database. This mapping is stored in an XML file called a “EDMX” (Entity Data Model XML) file.

You use Entity Framework to query the database and retrieve data, which is then materialized as objects in your application.

You can modify the objects in your application and use Entity Framework to persist those changes back to the database.

Overall, Entity Framework simplifies data access by providing a higher-level abstraction over the database, allowing you to work with data using familiar object-oriented concepts.

EDM

An Entity Data Model (EDM) is a conceptual model that describes the structure of a database in terms of the entities, relationships, and actions that can be performed on it. An EDM is typically used to design a database that will be used by an application.

An Entity Data Model XML (EDMX) file is an XML file that defines an EDM and its mapping to a database. It is a combination of three things:

A conceptual model: This is a high-level view of the data and the relationships between entities. It is defined using Entity Data Model (EDM) concepts such as entities, properties, and relationships.

A storage model: This is a low-level view of the database schema, including tables, columns, and keys. It is used to map the conceptual model to the database schema.

Mappings: These define how the conceptual model maps to the storage model. They specify how the entities and properties in the conceptual model map to tables and columns in the storage model.

When you use Entity Framework to query the database, the EDMX file is used to translate your queries into the appropriate SQL statements for the database. When you update the data in your application, the EDMX file is used to translate your updates into the appropriate SQL statements for the database.

Overall, the EDMX file is a key part of Entity Framework and plays a central role in mapping between the conceptual model and the database schema, as well as in translating queries and updates between the two.

Decorators

In Entity Framework, decorators are attributes that you can apply to your domain model classes and properties to provide additional information to Entity Framework. Decorators are used to specify various aspects of how Entity Framework should map your domain model classes and properties to the database.

Here are some examples of decorators that are commonly used in Entity Framework:

[Table]: This decorator is used to specify the table in the database that a class should be mapped to.

[Column]: This decorator is used to specify the column in the database that a property should be mapped to.

[Key]: This decorator is used to specify the primary key column for a table.

[ForeignKey]: This decorator is used to specify a foreign key column that is used to create a relationship between two tables.

[NotMapped]: This decorator is used to specify that a property should not be mapped to a column in the database.

Decorators are just one way to configure the mapping between your domain model classes and the database in Entity Framework. You can also use a separate XML file or code-based configuration to specify the mapping.

XML Configuration

To use a separate XML file for mapping in Entity Framework, you will need to create an EDMX file and specify the mapping between your domain model classes and the database schema in the Mappings section of the file. You can then use Entity Framework to read the EDMX file and use the mapping information it contains to query and update the database.

Overall, using a separate XML file for mapping can be a convenient way to specify the mapping between your domain model classes and the database in Entity Framework, especially if you have a complex mapping that cannot be easily expressed using decorators or code-based configuration.

EDMX file example

Here is an example of an EDMX file that maps a simple domain model to a database:

<?xml version="1.0" encoding="utf-8"?>
<edmx:Edmx Version="4.0" xmlns:edmx="http://schemas.microsoft.com/ado/2009/11/edmx">
  <edmx:Designer>
    <edmx:Connection>
      <DesignerInfoPropertySet>
        <DesignerProperty Name="MetadataArtifactProcessing" Value="EmbedInOutputAssembly" />
      </DesignerInfoPropertySet>
    </edmx:Connection>
    <edmx:Options>
      <DesignerInfoPropertySet>
        <DesignerProperty Name="ValidateOnBuild" Value="true" />
        <DesignerProperty Name="EnablePluralization" Value="True" />
        <DesignerProperty Name="IncludeForeignKeysInModel" Value="True" />
      </DesignerInfoPropertySet>
    </edmx:Options>
    <edmx:Diagrams />
  </edmx:Designer>
  <edmx:Runtime>
    <edmx:ConceptualModels>
      <Schema Namespace="ExampleModel" Alias="Self" xmlns:annotation="http://schemas.microsoft.com/ado/2009/02/edm/annotation" xmlns="http://schemas.microsoft.com/ado/2009/11/edm">
        <EntityType Name="Customer">
          <Key>
            <PropertyRef Name="Id" />
          </Key>
          <Property Name="Id" Type="Int32" Nullable="false" annotation:StoreGeneratedPattern="Identity" />
          <Property Name="Name" Type="String" MaxLength="50" Unicode="true" FixedLength="false" />
          <Property Name="Email" Type="String" MaxLength="255" Unicode="true" FixedLength="false" />
        </EntityType>
        <EntityType Name="Order">
          <Key>
            <PropertyRef Name="Id" />
          </Key>
          <Property Name="Id" Type="Int32" Nullable="false" annotation:StoreGeneratedPattern="Identity" />
          <Property Name="Total" Type="Decimal" Precision="18" Scale="2" />
          <Property Name="OrderDate" Type="DateTime" />
          <Property Name="CustomerId" Type="Int32" Nullable="false" />
          <NavigationProperty Name="Customer" Relationship="ExampleModel.CustomerOrder" FromRole="Order" ToRole="Customer" />
        </EntityType>
        <Association Name="CustomerOrder">
          <End Type="ExampleModel.Customer" Role="Customer" Multiplicity="1" />
          <End Type="ExampleModel.Order" Role="Order" Multiplicity="*" />
        </Association>
      </Schema>
    </edmx:ConceptualModels>
    <edmx:Mappings>
      <Mapping Space="C-S" xmlns="http://schemas.microsoft.com/ado/2009/11/mapping/cs">
        <EntityContainerMapping StorageEntityContainer="ExampleModelStoreContainer" CdmEntityContainer="ExampleModelContainer">
          <EntitySetMapping Name="Customers">
